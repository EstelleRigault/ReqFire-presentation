Introduction to the Use Case Methodology

ReqFire is a SaaS solution for creating use case based requirements models.

What is Use Case Analysis?

  • Use Case Analysis is a way of organising requirements so they become easier to understand
  • 'Formal' methodology - i.e. it has a specific format
  • Analysts and Developers know what to expect

What jobs is the Requirements Model doing?

  • Capture the complete list of requirements
  • Provide enough detail so that developers do not guess or assume what to build
  • Allow the Business to understand and commit to the Requirements
  • Identify and manage changes

Why does Use Case Analysis work?

Use Case Analysis is a simple way to conceptualise your whole model.

  • It uses a ‘story’ format – characters (Actors) and narrative (Use Cases) and leverages the ‘Narrative Paradigm’.
  • It provides a mental framework on which the detail can hang.
  • It uses diagrams to prime the reader before they see the detail.
  • It has rules to determine if the stories are detailed enough.
  • It separates the narrative flow from the rules and conditions.

What do we analyse and model

In UC analyse we focus on the Use Case and objects that are in that UC. Here are the objects we use:

  • Actors
  • Business Objects
  • Packages
  • Use Cases
  • Interfaces
  • Rules
  • Forms
People (and external Systems) that want to do something directly with the System.
Business objects whose state and data is managed or used by the system.
Logical groupings of functions relating to Use Cases
Stories describing in detail the Actors interactions to achieve a specific goal. Includes:
  • Flows: – paths through a Use Case, usually only one ‘Main Flow’, but there can be ‘Alternative Flows’.
  • Steps: Steps in a flow (viewable in activity diagram).
An interface to the System, can be a GUI/Web interface, an email format, a SMS etc.
Business rules applied during Use Cases.
A specific data exchange definition, maybe a web service or a messaging format.

Case Example: Adding a booking form

To demonstrate the app we are going to create a Use Case Package based off a client brief.

Client briefing

An existing client has asked us to scope and estimate the addition of a booking form on their website and members portal.

From a quick phone call you gather the following information:

  • Your client wants a way for people to book consultations.
  • People should be able to see what times are available before making a booking.
  • Members can book a time from their profile.
  • They prefer to keep the existing interfaces and pages.
  • People who are not members cannot book sessions.
  • People receive a notification once they have booked a session.

“When we launch the consultation booking feature can we change the header bar part at the top of the page to be more teal but with a gradient that blends slightly into brown but not too dark and more like turquoise meets light green brushed with a blueish tinge but undeniably teal”

Using your previous work and IP

We have previously done work with the client and already have their applications modelled. So the best place to begin is loading in our existing model from our Library.

The library feature is found on your Home page in Recent projects.

  • Click on Library.
  • To import our past work click on the download icon in Basic Web Membership
  • Now the Web Membership specifications for the client are in our current project.
  • This includes our interfaces, forms, rules, and use cases.

You can use the Library feature to import components from other models. You can also import any similar work from other projects or from a repository of generic packages. This is especially helpful for situations like our example web membership package.

What's in the package?

To see what was in the package we just imported navigate to the Model tab in the app. Here you will understand the previous specifications and objects to reuse in our project.

This is a good way to store IP within the company and make it easier for people to pick up and understand projects in a standard way.

Gather and Analyse Requirements

In this step we are going to load in our previous work with the client. Then we will turn the phone call with the client into traceable requirements.

Gather Requirements

Gathering requirements allows you to take advantage of the traceability and approval features in ReqFire.

  • Navigate to the Gather Tab in the app.
  • Start by adding in the client's requirements.
  • If it helps start by dragging the Requirement blocks into a hierarchy.
  • One approach is to organise requirements into topics, items and details.

Analyse Requirements

It's now time to link the analyse the requirements. There are three main goals at this stage:

  1. 1.

    Link the Basic Requirements to existing objects. E.g. The right interface for the requirement.

  2. 2.

    Add new objects to meet the requirements. E.g. New website functionality requires new Business Rules.

  3. 3.

    Get a sense if you have collected enough requirements. E.g. If something could be clarified by a requirement.

Goals 1 and 2 can be achieved by selecting the icons next to each Basic Requirement and then adding or searching the object. By the end of this process all requirements should be linked with at least one object.
